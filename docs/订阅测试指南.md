# TokenBoss 订阅 & 配额测试指南（V3 / newapi-as-truth）

> 跨过支付，本地手动测试 trial / 付费 / 过期降级全流程的命令集。
> Windows PowerShell 友好（不依赖 bash heredoc / `\` 续行）。

---

## 架构速览

V3 起，**newapi 完全托管订阅**——TokenBoss 不再保留任何"用户当前 plan / 何时到期 / 每日额度"的本地状态。注册时调一次 bind Trial、付费时调一次 bind Plus，剩下的（订阅过期、自动降组到 default、每日 quota 重置）全由 newapi 自己处理。

```
┌──────────────┐  Bearer sk-xxx          ┌──────────────────┐
│  client      │ ──────────────────────► │  TokenBoss       │
│  (curl/IDE)  │                         │  /v1/chat/...    │
└──────────────┘                         │                  │
                                         │  • sk → user 反查  │
                                         │  • 拉 newapi.group │
                                         │    决定 trial/默认 │
                                         │    模型改写到 eco  │
                                         │  • 透传到 newapi   │
                                         └────────┬─────────┘
                                                  │
                                         ┌────────▼─────────┐
                                         │  newapi          │
                                         │  • 订阅模块       │
                                         │  • bind 决定 group + quota │
                                         │  • 自动每日重置   │
                                         │  • 自动到期降组   │
                                         └──────────────────┘
```

| newapi `group` | 来源 | 模型 | TokenBoss 模型改写 |
|---|---|---|---|
| `trial` | 注册自动 bind Trial | $10 / 1 天 / 不重置 | 任何非 eco → `eco` |
| `plus` | 付费 webhook bind Plus | $30/天 / 每天重置 / 28 天 | 不改写 |
| `super` | 付费 bind Super | $80/天 / 每天重置 / 28 天 | 不改写 |
| `ultra` | 付费 bind Ultra | $720/天 / 每天重置 / 28 天 | 不改写 |
| `default` | 订阅过期后 newapi 自动回退 | 由 newapi 后台 default 组配置 | 任何非 eco → `eco` |

**数据流**：
- 当前订阅 / 余额 / 到期 → `newapi.listUserSubscriptions` (live) → `/v1/buckets`
- 钱包余额（订阅之外的钱）→ `newapi.getUser.quota - used_quota` → `/v1/me.balance`
- 用量统计 → `newapi.getLogs`

**TokenBoss DB 里**：只剩 `userId / email / passwordHash / newapiUserId / newapiPassword`。`plan / subscriptionStartedAt / subscriptionExpiresAt / dailyQuotaUsd / quotaNextResetAt` 这些列**保留 schema 但不再写**（兼容 V2 老数据）。

## 准备工作

### 1. 配 newapi plan_id 映射

在 newapi 后台 → 订阅管理 → 看每个套餐左侧的 `#ID` 列。把 4 个 id 填到 `backend/.env.local`：

```env
NEWAPI_PLAN_ID_TRIAL=2
NEWAPI_PLAN_ID_PLUS=1
NEWAPI_PLAN_ID_SUPER=3
NEWAPI_PLAN_ID_ULTRA=4
```

> 顺手在 newapi 后台确认 4 个套餐的 **"升级分组"** 都填了对应 group 字符串（plus/super/ultra/trial）。空字符串会让 bind 不改 group，整个 V3 链路就断了。

### 2. 起 dev server

```powershell
cd D:\openclaw\backend
npm run dev   # 另起一个终端窗口持续跑
```

下面所有命令都假设 `cd D:\openclaw\backend` 已经做过。

### 3. 准备测试用户

从 dashboard 注册一个用户，复制 default key。下文用 `<KEY>` 占位，`<USER_ID>` 占位 TokenBoss userId（看 dashboard URL 里那个 `u_xxx`，或查 `users` 表）。注册成功后这个用户已经被自动 bind 到 Trial（你应该能在 newapi 后台 → 订阅管理 → 用户订阅看到他的 trial 记录）。

## 1. 工具脚本

| 脚本 | 用途 |
|---|---|
| `scripts/grant-plan.ts <userId> <trial\|plus\|super\|ultra\|default>` | 跨过支付直接切换订阅；先 invalidate 现有订阅再 bind 新的 |
| `scripts/probe-newapi-bind.mjs <newapiUserId>` | 验证 bind API 接通：拉套餐列表 + bind Trial + diff |

所有脚本都从 `.env.local` 读 newapi 配置，不需要额外参数。

## 4. 场景三：每日 quota 重置 / 订阅过期

V3 中**这俩 TokenBoss 不管**——newapi 内部按 plan 的 `quota_reset_period` 自动重置：
- Trial：`never`（$10 一次性，1 天到期）
- Plus / Super / Ultra：`daily`（每天回到套餐的 total_amount）

**重置时间**是 UTC 0:00（北京时间 08:00）**全服同步**，不是 per-user 24h 循环。 dashboard 的"今 / 明 HH:MM 重置"文案直接显示 newapi 给的 `next_reset_time`。

**订阅过期**也由 newapi 自己处理：
- 套餐 `end_time` 到期 → newapi 自动 `status=cancelled`
- newapi 自动把用户 `group` 回退到 `default`
- TokenBoss 那边什么都不用做，下次 `/v1/buckets` 拉 → 看到无 active sub → dashboard 显示 default

要观察过期效果，进 newapi 后台 → 订阅管理 → 用户订阅 → 编辑某条订阅，把 `end_time` 改到过去，然后看：
- newapi 自己把那条 sub 标 cancelled
- newapi 自己把用户 group 改回 default
- 刷新 dashboard → 没有订阅卡，只有钱包余额

## 7. 常见问题

### 注册返回 200 但 newapi 那边用户 group 还是 default
没配 `NEWAPI_PLAN_ID_TRIAL`，bind 步骤被跳过（authHandlers 会在 backend 日志里打 warning：`NEWAPI_PLAN_ID_TRIAL not configured — skipping trial bind ...`）。在 `.env.local` 加上重启即可。

### 付费 settle 200 但 newapi group 没变
没配对应的 `NEWAPI_PLAN_ID_<PLAN>`，paymentWebhook 的 bind 步骤被跳过。同样在 `.env.local` 加上重启 backend。可以手动跑 `npx tsx scripts/grant-plan.ts <USER_ID> plus` 补救。

### bind 调用 200 但 newapi 用户 group 没变
检查 newapi 后台 → 订阅管理 → 对应套餐的"升级分组"字段是不是空。空字符串等于"不升级"。

### 用户付费后还在扣 trial 额度
这是早期 V3 的 bug——bind 不会自动 invalidate 旧 trial。现已修复：付费 webhook + grant-plan 都会先 list active subs 再 invalidate，最后 bind 新套餐。

### 自然过期 dashboard 还显示套餐
应该不会——`/v1/buckets` 是从 newapi 实时拉，无缓存。如果看到滞后，强刷一下 `Ctrl+Shift+R`，或者检查 newapi 那条 sub 的 `status` 是不是确实变成了 `cancelled`/`expired`。

### `Invoke-WebRequest : 无法绑定参数"Headers"`
PowerShell 5 把 `curl` aliased 到 `Invoke-WebRequest`。用 `curl.exe` 显式调真正的 curl。

### 调 chat 200 但 `choices: []`
newapi 渠道不可用。检查 newapi 后台对应 group（trial/plus/...）是不是挂了 chat 用到的渠道。

## 8. 套餐价格 / 时长配置

改 `backend/src/lib/plans.ts`：

```ts
export const PLANS = {
  plus:  { displayName: "Plus",  priceCNY: 288,  durationDays: 28, group: "plus"  },
  super: { displayName: "Super", priceCNY: 688,  durationDays: 28, group: "super" },
  ultra: { displayName: "Ultra", priceCNY: 1688, durationDays: 28, group: "ultra" },
};
```

> `durationDays` 在 V3 下**仅做向前端展示用**（"4 周套餐"文案）；真实订阅时长完全由 newapi 套餐里的 `duration_value + duration_unit` 决定。两边要保持一致，否则前端显示和实际到期不符。

测试期想用 ¥10 / ¥20 / ¥30 跑端到端付款，**用 env 覆盖**不动代码：

```env
PLAN_PRICE_PLUS_CNY=10
PLAN_PRICE_SUPER_CNY=20
PLAN_PRICE_ULTRA_CNY=30
```

测完删掉这三个 env 重启即恢复 ¥288 / ¥688 / ¥1688。

### 真要改每日额度 / 重置周期 / 升级 group 怎么办

去 **newapi 后台 → 订阅管理 → 编辑对应套餐**：
- `total_amount`：每日（或周期）总额度，单位 newapi 内部刻度（500,000 = $1）
- `quota_reset_period`：daily / weekly / monthly / never
- `upgrade_group`：bind 时给用户切到哪个 group 字符串
- `duration_value` + `duration_unit`：订阅时长

改完**对新订阅立即生效**，对已有订阅不变。要让现有用户也享受新参数，让他们重新 bind（`grant-plan.ts <USER_ID> <plan>`）。

## 充值 (topup) 自助下单 e2e 测试

### 准备

- 后端环境已配 `XUNHUPAY_APPID` / `XUNHUPAY_APPSECRET` 或 `EPUSDT_BASE_URL` / `EPUSDT_PID` / `EPUSDT_SECRET` 任一
- 后端环境已配 `NEWAPI_BASE_URL` / `NEWAPI_ADMIN_TOKEN`
- 测试账户已 register + emailVerified
- 起一个 `npm run dev` (backend) + `npm run dev` (frontend)

### 场景 1 — 支付宝 ¥1 充值（PC 扫码）

1. 在 PC 浏览器登录 → /console → 点 hero 上的 "充值额度"
2. 选 支付宝 + 自定义 `1` → 去付款 ¥1
3. /billing/orders/<id> 应显示扫码二维码
4. 用支付宝 app 扫码完成 ¥1 支付
5. 等 ≤60s，OrderStatus 应翻成 "支付成功 · $1 已加到余额"，3s 后自动跳回 /console
6. /console 上 Agent 余额 应 +$1
7. newapi admin 后台 → 用户 → quota 应 +500_000

### 场景 2 — USDT-TRC20 $1 充值（PC 新窗口）

1. 同入口 → 选 USDT-TRC20 + 自定义 `1`
2. 去付款 $1 → 应在新窗口打开 epusdt 收银台
3. 用 TRON 钱包付 ≈$1 USDT
4. 收银台跳回我们的 /billing/success?orderId=...
5. 同场景 1 后续验证：余额 +$1 + newapi quota +500_000

### 场景 3 — 移动端支付宝 H5 直跳

1. 手机 Safari / Chrome 登录 → /billing/topup
2. 选 支付宝 + 默认 ¥50 → 去付款 ¥50
3. 应直接跳支付宝 app 完成支付
4. 支付宝跳回 /billing/success?orderId=... → OrderStatus → "支付成功 · $50 已加到余额"

### 场景 4 — 兑换码入口仍然可用

1. 登录 → /billing/topup → 底部 "已有兑换码？" 链接
2. RedeemCodeModal 弹出 → 输管理员铸的 redemption code → 应 +$X 到余额

### 场景 5 — 活跃订阅者可充值

1. 用 `scripts/grant-plan.ts <email> plus` 给账户绑 Plus 套餐
2. 登录 → /billing/pay?plan=plus → lockout 页应显示 "+ 加买充值额度" 链接
3. 点链接进 /billing/topup → 完整完成 ¥1 充值
4. 余额 = Plus 当日 cap + $1 充值

### 场景 6 — 重复 webhook 投递不重复加额度

1. 完成场景 1 一笔 ¥1 充值，记下 orderId
2. 手动用 curl 给 /v1/billing/webhook/xunhupay 重复投递相同的 webhook 体（带正确签名）
3. 余额应保持 +$1，不变成 +$2
4. backend log 应 NOT 显示第二条 'topup credited' 行 — 第二条 webhook 在 markOrderPaidIfPending 步被原子性拦掉，返回 false，所以 applyTopupToUser 不会再次调用

### 场景 7 — newapi 失败时落 settleStatus=failed

1. 把 NEWAPI_ADMIN_TOKEN 临时改错 → 启动 backend
2. 走完整 ¥1 支付宝充值流程
3. 网关回调到 webhook → settle order paid → mint redemption fails
4. SQLite 查 `SELECT orderId, status, settleStatus FROM orders WHERE userId='...' ORDER BY createdAt DESC LIMIT 1;` 应是 `(...,'paid','failed')`
5. backend log 应有 `[webhook/xunhupay] createRedemption failed { orderId, userId, topupAmountUsd, errorMessage, channel }` 行
6. 修回 NEWAPI_ADMIN_TOKEN，跑 `scripts/grant-bucket.sh <email> topup 1` 兜底加额度（人工修复路径）
