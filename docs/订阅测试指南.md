# TokenBoss 订阅 & 配额测试指南

> 跨过支付，本地手动测试免费/付费用户全流程的命令集。
> Windows PowerShell 友好（不依赖 bash heredoc / `\` 续行）。

---

## 架构速览

```
┌──────────────┐  Bearer sk-xxx          ┌──────────────────┐
│  client      │ ──────────────────────► │  TokenBoss       │
│  (curl/IDE)  │                         │  /v1/chat/...    │
└──────────────┘                         │                  │
                                         │  • sk → user 反查  │
                                         │  • free 用户改写   │
                                         │    model 到 eco    │
                                         │  • 透传到 newapi   │
                                         └────────┬─────────┘
                                                  │
                                         ┌────────▼─────────┐
                                         │  newapi          │
                                         │  • 配额扣减       │
                                         │  • 用户 group     │
                                         │    决定渠道白名单  │
                                         └──────────────────┘
```

| 用户类型 | newapi `quota` | newapi `group` | TokenBoss 模型改写 |
|---|---|---|---|
| free（注册默认） | $10 一次性 | `free` | 任何非 eco 模型 → `eco` virtual |
| Plus | $30/天，cron 重置 | `plus` | 不改写，user 选什么模型就用什么 |
| Super | $80/天 | `super` | 不改写 |
| Ultra | $720/天 | `ultra` | 不改写 |

数据流：
- 用量统计 → 直接查 `newapi.getLogs`
- 余额（剩余）→ 直接查 `newapi.getUser.quota`
- 订阅状态（plan/expiresAt）→ TokenBoss `users` 表

## 1. 工具脚本

| 脚本 | 用途 |
|---|---|
| `scripts/grant-plan.ts <userId> <plus\|super\|ultra\|free>` | 跨过支付直接激活/降级订阅 |
| `scripts/run-cron.ts` | 手动触发每日重置 cron（活跃订阅刷 quota，过期订阅降级到 free） |
| `scripts/expire-plan.ts <userId>` | 把 `subscriptionExpiresAt` 推到昨天，方便测过期降级 |
| `scripts/backfill-key-index.ts` | 批量把现有 newapi token 写到 `api_key_index`（仅在迁移历史数据时用一次） |

所有脚本都从 `.env.local` 读 newapi 配置，不需要额外参数。

## 2. 场景一：Free 用户行为

**预期：所有 sk-xxx 调用都被静默改写到 eco 渠道。**

```powershell
# 调 opus（应被改写到 eco virtual → newapi 的 ecoTiers 主选）
curl.exe -sS http://localhost:3000/v1/chat/completions `
  -H "Authorization: Bearer <KEY>" `
  -H "Content-Type: application/json" `
  -d "{\"model\":\"claude-opus-4.7\",\"messages\":[{\"role\":\"user\",\"content\":\"hi\"}],\"max_tokens\":50}"
```

预期：
- HTTP 200
- response 里 `model` 字段是 `gpt-5.4-mini-xxx`（newapi 解析后的具体快照），**不是** `claude-opus-4.7`
- 后端 dev 日志能看到 `[free-rewrite] claude-opus-4.7 → eco`

刷 `/console`，"已用" 应该有数（4 位精度），"余额" 从 $10 往下掉。

## 3. 场景二：激活付费订阅

```powershell
npx tsx scripts/grant-plan.ts <USER_ID> plus
```

脚本会打印 BEFORE/AFTER：
- `users.plan` free → plus
- `users.dailyQuotaUsd` null → 30
- `users.subscriptionExpiresAt` 30 天后
- newapi.quota $10 → $30
- newapi.group free → plus

刷 dashboard：
- hero 卡片显示 plus 套餐
- "今日额度" 显示 `$0.0000 / $30`
- bucket 列表 skuType=`plan_plus`

调 chat，验证**不再被改写**：
```powershell
curl.exe -sS http://localhost:3000/v1/chat/completions `
  -H "Authorization: Bearer <KEY>" `
  -H "Content-Type: application/json" `
  -d "{\"model\":\"claude-opus-4.7\",\"messages\":[{\"role\":\"user\",\"content\":\"hi\"}],\"max_tokens\":50}"
```

response 里 `model` 应该是 `claude-opus-...`（如果 newapi `plus` 组没挂 opus 渠道，会拿到 `model_not_found` 错误——但**没有静默改写**，证明分流逻辑正确）。

## 4. 场景三：24h 配额重置

每个付费用户有自己独立的 24h 重置周期（不是日历日 0:00 全员重置），从激活那一刻起算 24h 一周期。后端 cron 默认每 60 分钟扫一次，重置那些 `quotaNextResetAt <= now` 的用户。

要测试，最快方式是手动把 `quotaNextResetAt` 改到过去。或者直接跑：

```powershell
# 跑一次 sweep，看现在有谁 due
npx tsx scripts/run-cron.ts
```

如果用户刚激活，`quotaNextResetAt` 大概是 24 小时后，跑 sweep 不会触发任何重置——这是对的（输出 `{ reset: 0, ... }`）。

要立即测重置效果，改一下数据库把窗口拉到过去：

```powershell
# 手动把 nextResetAt 推到 1 分钟前（跟 expire-plan 同款思路）
node -e "const D=require('better-sqlite3');const d=new D('data/tokenboss.db');d.prepare('UPDATE users SET quotaNextResetAt = ? WHERE userId = ?').run(new Date(Date.now()-60000).toISOString(),'<USER_ID>');console.log('done');d.close()"

# 再跑 sweep
npx tsx scripts/run-cron.ts
```

预期输出：
```
[cron] result: { reset: 1, expired: 0, failed: 0 }
```

newapi.quota 被重新设回 dailyQuotaUsd × 500_000；user.quotaNextResetAt 自动 +24h。

**配置 cron 间隔**：在 `.env.local` 加 `QUOTA_CRON_INTERVAL_MINUTES=5` 让 dev server 每 5 分钟扫一次（默认 60）。改完重启 `npm run dev` 生效。

## 5. 场景四：订阅过期降级

```powershell
# 把 expiresAt 推到昨天
npx tsx scripts/expire-plan.ts <USER_ID>

# 再跑 cron
npx tsx scripts/run-cron.ts
```

预期：
```
[cron] result: { reset: 0, expired: 1, failed: 0 }
```

效果：
- `users.plan` plus → free
- `users.subscriptionExpiresAt` → null
- `users.dailyQuotaUsd` → null
- newapi.quota → 0
- newapi.group → free

刷 dashboard：用户回到 free 状态，余额 $0（quota 被清零了，注意——降级**不会**自动再给 $10，那是注册时的一次性发放）。

## 8. 常见问题

### `Invoke-WebRequest : 无法绑定参数"Headers"`
PowerShell 5 把 `curl` aliased 到 `Invoke-WebRequest`，参数语义不一样。用 `curl.exe` 显式调真正的 curl。

### `Unexpected end of JSON input` 或 `loginUser: empty response from newapi (status 429)`
newapi 限流。已经修过——newapi session 缓存 5 分钟。如果还是看到 429，说明缓存被频繁 invalidate 了，或者你在外面写脚本绕过了缓存。等 1 分钟再试。

### `npx tsx -e "..."` 多行命令在 PowerShell 起 REPL
PowerShell 处理 `-e` 多行串经常出错。把 inline 串改成脚本文件跑（`npx tsx scripts/xxx.ts`）。

### 调 chat 200 但 `choices: []`
newapi 返回了 OpenAI shape 但渠道不可用。检查：
1. newapi 后台对应 group（free/plus/...）是否挂了 chat 用到的渠道
2. router-tiers.json 的 ecoTiers 配的模型是不是你 newapi 实际有的

### 余额、已用显示成 $10.00 / $0.00
显示精度问题。已经把所有 USD 字段改成 4 位（`.toFixed(4)`），单条 record 用 6 位（`.toFixed(6)`）。如果还看到老精度，刷新一下浏览器（Vite HMR 偶尔会漏）。

## 9. 套餐价格 / 配额配置

改 `backend/src/lib/plans.ts`：

```ts
export const PLANS = {
  plus:  { displayName: "Plus",  priceCNY: 288,  dailyQuotaUsd: 30,  durationDays: 30, group: "plus"  },
  super: { displayName: "Super", priceCNY: 688,  dailyQuotaUsd: 80,  durationDays: 30, group: "super" },
  ultra: { displayName: "Ultra", priceCNY: 1688, dailyQuotaUsd: 720, durationDays: 30, group: "ultra" },
};

export const FREE_TIER = {
  initialQuotaUsd: 10,
  group: "free",
};
```

改完之后：
- 新用户注册自动走新值
- 已激活的旧订阅**不会自动跟随**——下次 cron 才把 quota 刷成新的 dailyQuotaUsd
- newapi 那边对应的 group 名字必须实际存在；建不建组不影响 quota 扣减，但影响 group→渠道映射
